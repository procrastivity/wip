# w2 — Project-type-aware bootstrap tooling (`wip setup *`)

Read `.wip/distillation/BRIEF.md` first. You are agent **w2**.

## CRITICAL framing (from the human)

You are **NOT** auditing or making the example repos consistent. Your job: identify
the **clearly-best baseline patterns** across the examples, then design **bootstrap
tooling/skills** that make it trivial to set a NEW or existing project up in that
style. Think `wip setup direnv` / `/wip:setup release` / `/wip:setup hygiene` —
commands that **detect project type and ask the right questions**, e.g.:

- `wip setup direnv` → detect PHP vs Python (uv) vs shell vs Node; if Docker present,
  ensure hadolint; wire flake.nix + .envrc accordingly.
- `/wip:setup release` → ask whether they want a changelog; wire cliff.toml + Makefile
  snippet + releasing doc (the `changelog-portable-stub` pattern).
- `/wip:setup hygiene` → detect file types, propose the right pre-commit hooks
  (shfmt/shellcheck for bash, ruff for python, etc.) + the AGENTS.md↔CLAUDE.md pointer.

## Your deliverable

1. **The canonical baseline**, distilled with evidence: what "good" looks like for
   flake/direnv, uv (python), Docker/hadolint, dependabot, pre-commit hygiene,
   changelog, and the AGENTS.md(canonical) ← CLAUDE.md(pointer) convention. Cite the
   xcind north-star and call out the prtend drift as the anti-pattern to prevent.
2. **A `wip setup` command family spec**: each subcommand's detection logic (how it
   sniffs project type), the questions it asks, what it writes, and idempotency/
   re-run behavior. Cover at least: `direnv`, `release`, `hygiene`, `agents` (the
   CLAUDE→AGENTS pointer), `dependabot`.
3. **Dual surface**: how each is BOTH a `wip` CLI subcommand AND a `/wip:setup <x>`
   Claude Code slash command. Recommend which logic is deterministic-script vs
   needs-an-agent (judgment).
4. **How the portable stubs fold in**: `changelog-portable-stub` and
   `direnv-session-loader` already exist as installers — should `wip setup` wrap/
   replace/vendor them? Recommend.
