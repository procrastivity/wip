# w5 — The `wip` CLI + `/wip:*` command grammar

Read `.wip/distillation/BRIEF.md` first. You are agent **w5**.

## Your deliverable

You own the **unifying surface** that makes "what's the state of things / what's next"
deterministic. You depend on the other four — design against their SEEDS in the BRIEF
and note where you assume something they may refine.

1. **The full command grammar**: a coherent verb/noun scheme for both the `wip` CLI and
   the `/wip:*` slash commands. Map every capability the other workstreams imply:
   - `wip status` / `wip next` — deterministic state + next action (consumes w1 lifecycle,
     w4's "what's next" source-of-truth stance).
   - `wip init` / `wip intake` — start an initiative; spec-generator vs handoff intake (w1).
   - `wip setup <direnv|release|hygiene|agents|dependabot>` — w2's bootstrap family.
   - `wip doctor` / `wip detect` — w3's feature-detection contract (is LDS/Diátaxis/
     changelog/prtend/Solo present, and where).
   - `wip extract` / `wip graduate` — w3's `.wip` → LDS pipeline.
   - `wip orchestrate` / spawn helpers — w4's actor playbooks.
2. **CLI vs slash-command split**: which commands are deterministic scripts (no LLM) vs
   which are agent-backed (`/wip:*` invoking judgment). Recommend a clear rule. Note the
   prtend principle ("CLI never calls an LLM; judgment lives in the skill") as a model.
3. **Distribution & composability**: how `wip` ships (single binary? bash like prtend/
   xcind? plugin like direnv-session-loader's `.claude-plugin`?), and how features are
   opt-in per the `wip.yaml` manifest. Recommend.
4. **The minimal v1 surface**: of everything above, what's the smallest shippable core
   that delivers "deterministic state of things"? Sequence the rest.
